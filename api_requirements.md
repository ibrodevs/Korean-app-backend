# API Requirements Document - Korean App Backend

## 1. Общая информация (Overview)
- **Что делает API?** API предоставляет эндпоинты для управления каталогом товаров (категории, бренды, продукты, вариации продуктов, атрибуты и их значения), сейчас есть все ендпоинты для получение данных но нет для создания и редактирования, удаления.
- **Кто будет использовать API?** веб-клиент будут использовать API как админ панель для тех которые будут заводить товары в  каталог. 
- **Почему нужен этот API?** Для обеспечения централизованного управления контентом и товарами, проведения быстрых CRUD-операций с сущностями электронной коммерции 
- **Модели/ресурсы:** `Category`, `Brand`, `Product`, `ProductVariant`, `ProductImage`, `Attribute`, `AttributeValue` (включая мультиязычные переводы).
- **Операции для моделей:** Полный CRUD (Create, Read, Update, Delete) для администраторов. 
- **Фильтры, сортировки, пагинация:** Обязательны пагинация для списков продуктов, сортировка (по цене, новизне и т.д.), и сложная фильтрация по атрибутам, ценам и брендам (с учетом интеграции Elasticsearch).
- **Вложенные объекты:** Возврат связанных объектов, например: `Product` → `Category`, `Product` → `Brand`, `Product` → `ProductVariant`, `ProductVariant` → `ProductImage` и `Attributes`.

## 3. Non-Goals
- **Какие функции не включены:** На данном этапе не реализуется сложная аналитика продаж, рекомендательные системы на базе AI и интеграции со сторонними ERP/CRM (1C, МойСклад).
- **Что точно не должно быть реализовано:** Управление корзиной, оформление заказов (Order) и платежные шлюзы, если задача строго ограничена текущим CRUD API для продуктового каталога.

## 4. Технические ограничения (Tech Constraints)
- **Технологии:** Django + Django REST Framework (DRF).
- **База данных:** PostgreSQL (в качестве основной реляционной БД) и Redis (для кэширования celery/интерфейсов). Проект докеризирован (`docker-compose.yml`).
- **Формат ответа:** JSON.
- **Аутентификация и авторизация:** JWT-токены (используется `rest_framework_simplejwt`), также предусмотрена интеграция OAuth2 (Google Client) для мобильных устройств.
- **Хостинг/Сервер:** Разворачивание посредством Docker, что подразумевает работу в контейнерах; ограничения по RAM/CPU согласно настройкам серверов/контейнеризации.

### Локальная разработка (Python version)
- Проект собирается и запускается в Docker на `python:3.10-slim` (см. `Dockerfile`).
- Для установки зависимостей локально используйте Python **3.10–3.12**.
- Python **3.13** для текущих пинов в `requirements.txt` не подходит: части зависимостей (например, `pandas==2.1.2`, `numpy==1.26.4`) не имеют готовых wheel под 3.13 и падают при сборке из исходников.

## 5. Критерии успеха (Success Criteria)
- API должно корректно поддерживать CRUD для всех основных моделей (`Category`, `Brand`, `Product`, `ProductVariant` и др.).
- Время отклика оптимизировано (оптимизация запросов, устранение N+1 проблем с помощью `select_related`/`prefetch_related` в ORM или использования Elasticsearch).
- Прохождение тестов: наличие unit- и integration-тестов, подтверждающих корректность работы бизнес-логики и эндпоинтов.
- Документация: Swagger/ReDoc генерируется автоматически через `drf-spectacular` и полностью описывает все эндпоинты, их параметры и возможные ответы.

## 6. Фазы (Phases)
- **Phase 1:** Создание базовых CRUD эндпоинтов (с сериализаторами и views) для моделей ресурсов каталога (`Category`, `Brand`, `Product`, сериализация вариантов и атрибутов).
- **Phase 2:** Настройка аутентификации, авторизации и добавление продвинутой фильтрации (внедрение Elasticsearch, django-filter).
- **Phase 3:** Написание автоматизированных тестов для эндпоинтов и генерация схемы API/документации (Swagger/ReDoc).
- **Phase 4:** Оптимизация производительности (кэширование через Redis, профилирование БД-запросов).

## 7. Правила агента (Agent Rules)
- **Always (Всегда):** Тестировать все эндпоинты локально перед деплоем / подтверждением готовности (писать и запускать тесты DRF).
- **Ask First (Спроси сначала):** Любые изменения в модели структуры базы данных (новые поля, удаление моделей) строго согласовывать с командой/пользователем.
- **Never (Никогда):** Не использовать небезопасные методы (например, открытый DELETE без авторизации и проверки прав администратора). Учитывать доступы при получении или мутации данных.
