Product Requirements Document: E-commerce Catalog API (GET-only)
1. Overview

    What (Что): Создать набор быстрых и удобных HTTP-эндпоинтов для чтения данных каталога (товары, категории, бренды, атрибуты) внешними клиентами (веб-сайт, мобильное приложение).

    Who (Кто): Frontend-разработчики и мобильные разработчики, которые будут использовать API для отображения каталога, а также конечные пользователи, ожидающие мгновенной реакции интерфейса.

    Why (Зачем): Обеспечить молниеносную загрузку страниц каталога, гибкую и точную фильтрацию товаров по множеству параметров (включая EAV-атрибуты) и стабильную работу при росте базы данных, заложив основу для будущего расширения функциональности.

2. Core Features (MVP для GET-запросов)

Мы реализуем только эндпоинты для чтения, но сделаем их максимально мощными.

    Получение списка товаров с расширенной фильтрацией
    Клиент сможет получать товары с пагинацией, применяя фильтры по основным полям (категория, бренд, цена, активность) и по динамическим атрибутам (например, "объем памяти", "цвет").

        Пример запроса: GET /api/v1/products/?category=laptops&brand=apple&price[min]=500&price[max]=1500&attr_ram=16&attr_color[]=space-gray&attr_color[]=silver

    Полнотекстовый поиск по каталогу
    Эндпоинт для поиска товаров по названию и описанию с учётом языка пользователя.

        Пример запроса: GET /api/v1/products/?search=игровой ноутбук

    Детальная карточка товара (Product Detail)
    Получение всей информации о конкретном товаре по его slug: основные данные, все варианты (SKU) с ценами и остатками, все изображения (общие и для каждого варианта), все глобальные атрибуты товара.

        Пример запроса: GET /api/v1/products/igrovoi-noutbuk-asus-rog/

    Навигационные фильтры (Faceted Search)
    Специальный эндпоинт, который не только возвращает товары, но и агрегированную информацию для построения умных фильтров: список доступных брендов, диапазон минимальной и максимальной цены, список возможных значений атрибутов с количеством подходящих товаров.

        Пример запроса: GET /api/v1/catalog-search/?category=laptops

    Получение справочной информации
    Эндпоинты для получения плоских справочников: дерево категорий со всеми переводами, список брендов.

        Пример запроса: GET /api/v1/categories/

3. Non-Goals (Что мы НЕ ДЕЛАЕМ в этом проекте)

Этот раздел критически важен для фокусировки на задаче .

    Мы НЕ реализуем никакие эндпоинты для создания, изменения или удаления данных (POST, PUT, PATCH, DELETE). Это strictly Read-Only API.

    Мы НЕ реализуем систему авторизации и аутентификации пользователей (JWT, OAuth, сессии). API будет открытым для чтения (public read-only) на первом этапе.

    Мы НЕ занимаемся интеграцией с платёжными системами, корзиной, заказами или личным кабинетом пользователя.

    Мы НЕ пишем сложную бизнес-логику, связанную с расчётом скидок, персональных цен или проверкой прав доступа к конкретному товару.

    Мы НЕ строим отдельную админ-панель для управления каталогом (она будет позже, с другими эндпоинтами).

5. Success Criteria

Чёткие, проверяемые критерии готовности API .

    Функциональность: Все перечисленные в разделе 2 эндпоинты реализованы и задокументированы в Swagger/OpenAPI.

    Фильтрация: Эндпоинт списка товаров позволяет фильтровать по category, brand, price[min], price[max], is_active, а также минимум по трём различным динамическим атрибутам (например, attr_ram, attr_storage, attr_color).

    Пагинация: Реализована курсорная пагинация для списка товаров, гарантирующая стабильную скорость на глубоких страницах. Ответ содержит токены next и prev.

    Поиск: Поиск по запросу search находит соответствующие товары по названию и описанию на нужном языке.

    Производительность: Время ответа для эндпоинта списка товаров (с фильтрацией по 3 параметрам и пагинацией) на тестовой базе в 100 000 товаров не превышает 300 мс.

    Контракт API: Формат ответов соответствует JSON:API спецификации или иному заранее утверждённому стандарту, включает мета-информацию о пагинации и ссылки (HATEOAS-like).

    Документация: Документация в Swagger UI содержит понятные примеры запросов и описания всех параметров фильтрации.

6. Implementation Phases

Пошаговый план с верификацией на каждом этапе.

    Фаза 0: Подготовка и базовая структура

        Действия: Установка и настройка Django REST Framework, django-filter, drf-spectacular. Создание базовых ViewSets и сериализаторов для Category и Brand.

        Верификация: Эндпоинты /api/v1/categories/ и /api/v1/brands/ возвращают корректные данные. Swagger-схема генерируется без ошибок.

    Фаза 1: Ядро каталога (Товары и варианты)

        Действия: Создание ProductViewSet и ProductVariantViewSet. Реализация эндпоинтов для детальной карточки товара и списка товаров с базовой фильтрацией (по категории и бренду). Настройка пагинации.

        Верификация: GET /api/v1/products/ возвращает список товаров. GET /api/v1/products/{slug}/ возвращает полную информацию о товаре, включая вложенные варианты и изображения. Пагинация работает (есть ссылки на следующую/предыдущую страницу).

    Фаза 2: Мощная фильтрация и поиск

        Действия: Интеграция django-filter. Добавление фильтров по цене (min_price, max_price), активности. Реализация кастомного фильтра для динамических EAV-атрибутов (парсинг параметров вида attr_{slug}). Подключение SearchFilter для поиска по переводам названия и описания.

        Верификация: Запрос GET /api/v1/products/?category=laptops&price[min]=500&price[max]=1500&attr_ram=16 возвращает только товары, соответствующие условиям. Поиск ?search=игровой находит нужные товары.

    Фаза 3: Фасетный поиск (Faceted Search)

        Действия: Создание отдельного эндпоинта catalog-search или доработка существующего для возврата агрегированных данных. В ответе, помимо товаров, возвращаются valueFacets (например, список брендов и их количество) и rangeFacets (мин/макс цена).

        Верификация: Запрос на GET /api/v1/catalog-search/?category=laptops в ответе содержит секцию facets с брендами, доступными в категории "Ноутбуки", и их количеством.

    Фаза 4: Оптимизация и финализация

        Действия: Профилирование запросов с помощью django-debug-toolbar или аналогичного инструмента. Оптимизация select_related и prefetch_related в ViewSet. Настройка кэширования для статичных данных (категории, бренды). Проверка и доработка OpenAPI-спецификации.

        Верификация: Все эндпоинты проходят тесты производительности. Документация в Swagger полная и понятная. Код покрыт базовыми тестами для проверки фильтров.

7. Agent Rules

Правила для принятия решений при разработке.

    Always (Всегда):

        Всегда использовать select_related и prefetch_related для оптимизации запросов к БД в ViewSet.

        Всегда документировать все параметры запроса (фильтры, пагинацию) с помощью drf-spectacular.

        Всегда возвращать ошибки в стандартизированном формате (например, {"error": {"code": "invalid_filter", "message": "..."}}).

        Всегда использовать курсорную пагинацию для списков, где ожидается много страниц.

        Всегда проверять входные параметры на корректность (например, что price[min] — это число).

    Ask First (Спросить перед действием):

        Перед изменением формата ответа (например, добавлением новых полей в корень JSON), который может сломать существующие клиенты.

        Перед внедрением сложной логики кэширования, которая может потребовать инвалидации.

        Если производительность запроса с кастомным фильтром по EAV-атрибутам вызывает сомнения и требует нетривиального решения (например, создания индекса или изменения подхода).

    Never (Никогда):

        Никогда не добавлять в этот проект эндпоинты для создания/изменения данных (POST/PUT/DELETE).

        Никогда не использовать PageNumberPagination для списка товаров, если не доказана её эффективность и нет требования к переходам по номеру страницы.

        Никогда не возвращать чувствительные данные (например, внутренние ID или служебные поля) в ответах API.

        Никогда не игнорировать ошибки валидации параметров запроса — всегда возвращать 400 Bad Request с пояснением.
